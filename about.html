The Vinyl Scout
Collection Add Audit

How it works · v21 · June 2026

A pocket catalog for a personal record collection.

The Vinyl Scout is a small, opinionated web app for tracking one person's vinyl LPs — what's owned, what condition it's in, and what each pressing is worth on the secondary market. No accounts, no third-party tracking, no clever automations that quietly delete data.

01What it is

A single-tenant catalog of physical records. The owner photographs covers in small batches, Claude identifies artist and title from each cover, and a short list of Artist - Title lines gets pasted into /seed.html to register the records. Every entry is editable on /audit.html.

The collection page itself — what you came from — is read-only: a searchable, genre-filterable list or gallery. Tap any record for the detail view.

02The stack

  • Frontend. Static HTML, CSS, and one JavaScript file. No framework, no build step. Served from Netlify's CDN.
  • Storage. A single Netlify Blobs store called records. One JSON blob per record, keyed by ID.
  • API. One Netlify Function (/api/records) that handles GET (list), POST (upsert by ID), and DELETE (single record by ID).
  • Source. Plain Git repo at github.com/snesbitt/vinyl-scout. Every change ships via git push; Netlify rebuilds in under a minute.
  • Authoring. Code and copy written collaboratively with Claude; deploys reviewed and run by the owner.

03The catalog record

Every record is one JSON object. Phase 1 fields are the originals; Phase 3 adds condition and pricing.

  • id — opaque 16-character hex string (e.g. rec_a1b2c3d4e5f60718).
  • artist, title — required strings.
  • year, genre — optional. Year is 1900–2100.
  • cover_url — optional image URL or inline data URL.
  • notes — optional free text.
  • condition — Goldmine grade (default VG). See below.
  • price_low, price_median, price_high — market prices in price_currency, optional. Low is the current cheapest listing; median and high are from the release's last-30-sales history.
  • price_last_sold — string date of the most recent completed sale (e.g. "Apr 23, 2026"), or null if the release has never sold.
  • copies_available — count of copies currently for sale, optional.
  • have_count, want_count — Discogs community ownership and wishlist counts, optional.
  • rating_avg, rating_count — Discogs community rating (0–5) and number of ratings, optional.
  • price_currency — ISO code (USD, EUR, GBP), optional.
  • price_updated_at — ISO timestamp of last refresh, optional.
  • discogs_release_id — cached release ID for fast refreshes, optional.

04Goldmine grading

Vinyl conditions follow the Goldmine standard, the lingua franca of record collectors and dealers. Each grade carries an implicit multiplier against a "Near Mint" reference price — used when the catalog later computes an asking price from market data.

Mint — sealed, never played, factory-perfect.

NM

×1.10
VG+

×0.15

Grades are edited per-record on /audit.html via the rightmost dropdown. New records default to VG.

05Roadmap

  • Phase 1 — complete. Vision-identified records, edit, delete, cover upload.
  • Phase 3 — complete. Condition tracking on every record (Goldmine grade, default VG, editable on Audit).
  • Phase 3.1 — shipped (v12 — v18). Live Discogs market data. Open any record's detail view and tap "Refresh from Discogs" to pull the current cheapest listing and the full sales-history Statistics block — range, median, last sold date, copies for sale, and the Have/Want community counts. One record per click; no automatic refresh, no batch.
  • Phase 3.2 — not started. A "pick the right pressing" picker on Audit, for when Discogs's first search result isn't the right release.
  • Phase 2 — deferred. Pressing-accurate metadata from Discogs (label, catalog number, country, pressing year). Useful but not urgent.
  • Phase 4 — parked. Listing & selling helpers. Far-future.

What Discogs can and can't tell us. The Discogs API exposes the current cheapest listing and the number of copies for sale right now. It does not expose historical completed-sale data (low, median, high, last-sold date) on any documented endpoint — Discogs has an open feature request from the community for exactly this. That data is, however, in the static HTML of every public release page, so on each refresh we run two requests in parallel: the API for live market state, and a fetch-and-parse of the release page's Statistics block for sales history and community counts. If Discogs ever changes their page layout or blocks the fetch, the Market block falls back gracefully to API-only data and a small debug payload lands in the browser console so we know to update the parser.

06What it won't do

Three things, by charter, the Vinyl Scout will never do, because a previous iteration learned them the hard way.

  • No bulk delete. Every delete is one record at a time, behind a confirm dialog.
  • No auto-dedup. The earlier version's dedup logic raced with a polling loop and wiped 29 records. That code path is permanently banned.
  • No background mutation. Read endpoints only read. Writes happen only on explicit user action.
